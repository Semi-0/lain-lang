## 1. Workshop-fast path (next ~3 weeks) — living command center

*Until the workshop is over, this section is the only plan that really matters. The phased roadmap below is for after.*

**Must do:**
- **A1** GC / disposal boundary — propagator-only dispose; scheduler disposes cells when unreachable.
- **A2** Minimal timestamp/version fix — enough so card edits in chain don’t break; avoid channel explosion (epoch or substitutive merge on demo path).
- **B1** Card rebuild without immediate destruction — build new network in neighbor cell, retract old premises, let GC clean up.
- **B2** Only the narrow hot-reload path you need — premise/dependency-driven successor networks for the one corridor you demo (TemporaryValueSet + PremisesSource; no need to fix all of incremental_compiler).
- **Protocol simplification only on the demo path** — simplify decoder layer (pattern matcher + generic operators) only where it blocks the demo.
- **Minimal persistence** — loading/saving demo cards so the workshop can “save and reload”; no general storage layer yet.
- **D1** Topology tracking — core for demonstrating multi-projection; needed so the demo can show the graph / projection structure.
- **D2** Metadata/annotation layer — necessary for composable visualization; interested entities + annotation layer so viz can be composed and modified.

**Should do only if time remains:**
- **C3** Semantic card model — only enough to keep the code coherent (no full propagator-like card yet).
- **E1** Syntax — only if the demo absolutely needs missing syntax; if so, pull it into the fast path (see Phase E note below).

**Do not do before workshop unless something explodes:**
- **C4** Full card-as-construct composition
- **E2** Extra reactive operators
- **E3** CORE-03 beyond very light notes
- Reflective card/UI ambitions (network-card vs execution-card split, meta UI)

**Fastest path (operational):**
1. **Stabilize one demo corridor** — one flow that runs end-to-end.
2. **Then** reduce friction on that corridor — rebuild, hot reload, protocol, persistence only where it hurts.
3. **Then** add only the visual or semantic pieces needed to make that corridor impressive.

## 4. Architecture refactor (checklist)

Ordered list of architecture refactor items (aligned with phases A–E). **Prioritized in Phase C:** C1–C4 (protocol on hot path, storage, semantic card model, card-as-construct).

1. **GC and temporal/version semantics** — propagator-only dispose; scheduler-driven cell GC; vector clock / epoch or substitutive merge for card timestamps.
2. **Card rebuild protocol** — build new network in neighbor cell, retract old premises, GC gradually (no immediate destruction).
3. **Hot reload / replacement semantics** — premise/dependency-driven successor networks; TemporaryValueSet + PremisesSource.
4. **Frontend/backend protocol simplification on the hot path** — simplify decoder layer using pattern matcher + generic operators; scope to demo path.
5. **Storage layer for persisted card content** — where and how card state is stored across reload.
6. **Semantic card model** — card as compositional unit, possibly propagator-like.
7. **Card-as-construct composition** — spawning and local composition; pass card/cell construct into neighbor.
8. **Topology efficiency** — progressive graph tracking (no full BFS per update).
9. **Topology metadata/annotation** — interested entities; annotation layer for visualization modifiers.
10. **Minimal syntax and reactive operators for demo** — branching, sugar, reactive_map/filter/reduce, zip, etc., as needed.
11. **Reflective network-card / execution-card split and meta UI** — strict separation; unified input API; multi-projectional card system; spawned card as projected card.

## 5. Dependency Summary

- **A1 (GC)** and **A2 (vector clock)** are foundational: they make safe gradual disposal and correct card timestamps possible.
- **B1 (card rebuild without immediate destruction)** and **B2 (hot reload through premise/dependency-driven successor networks)** depend on A1/A2 and directly improve demo behavior.
- **C1–C4** (card model and persistence) cover protocol simplification on hot path, storage, semantic card model, and card-as-construct composition.
- **D1–D2** (topology and introspection) are required for the demo: topology tracking is core for multi-projection; metadata/annotation layer is necessary for composable visualization.
- **E1–E3** (language surface and reactive primitives) cover syntax and reactive operators needed by the demo, with CORE-03 experiments kept capped (E3).
